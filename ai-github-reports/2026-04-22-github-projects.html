📌 一、这是什么

Edict 是一个基于 OpenClaw 的 Multi-Agent 编排系统。它把中国古代三省六部制（公元 606 年——1901 年，运行约 1300 年的帝国行政体系）直接映射为 AI Agent 的协作架构。

整套系统共 12 个 Agent，各有其"职能"，通过 Redis Streams 事件总线通信，配备实时看板、完整审计链、热切换模型等生产级功能。

🔑 核心洞见：现代 Multi-Agent 框架大多是「做完就交」——没有质量关卡。三省六部制最大的制度创新，是把"门下省封驳"引入 AI 工作流，让产出在送到执行层之前必须先过一道强制审核。

六大核心技术亮点

  • EventBus 事件总线：Redis Streams 发布/订阅，服务间解耦通信
  • Outbox Relay：事务性 Outbox 模式，保障事件可靠投递（至少一次语义）
  • 状态机审计：严格生命周期状态转换 + 完整审计日志，每次流转都有记录
  • 并行调度引擎：Dispatch Worker 支持指数退避重试 + 资源锁
  • DAG 编排器：Orchestrator 基于有向无环图的任务分解与依赖解析
  • Agent 思考可视化：实时展示 thinking 过程、工具调用、返回结果

💡 五、为什么值得关注

Multi-Agent 系统的核心挑战从来不是"让 Agent 能跑起来"，而是如何让它在复杂任务下输出可靠结果。Edict 的三个核心洞见：

  • 制度性审核 > 提示词优化：与其反复调 prompt 希望 LLM 输出质量更高，不如在架构层引入强制的"门下省封驳"机制——输出不达标就打回重做。
  • 可观测性是生产必需：没有实时看板的 Multi-Agent 系统就像黑箱，出错时无从排查。军机处看板将所有 Agent 的状态、Token 消耗、任务流转全部可视化。
  • 人机协同 > 全自动：任务随时可叫停、干预、恢复，把人始终放在决策链路上，而不是"把所有事交给 AI 然后祈祷"。

🏛️ 古代三省六部制能运行 1300 年，靠的是「制度」而非「依赖某个聪明人」。把这套分权制衡的设计移植到 AI Agent 架构，或许是让 Multi-Agent 系统真正走向可靠的一条路。
