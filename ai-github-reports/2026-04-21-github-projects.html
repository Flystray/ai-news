⚡ SGLang

开源最快的 LLM 推理服务框架 — 用 RadixAttention 和多进程架构，把大模型推理性能压到极致

⭐ 26,100 stars 🍴 5,500 forks 📝 Python 📄 Apache-2.0 🔗 sgl-project/sglang
inference cuda deepseek attention blackwell diffusion multimodal llm-serving

SGLang 解决的核心问题是：如何让大模型在生产环境中又快又省。它通过自研的 RadixAttention 前缀缓存、多进程解耦架构和 CUDA Graph 优化，把 LLM 推理的吞吐量和延迟同时压到极致——目前是开源界最快的推理引擎之一，在 DeepSeek、Llama、Qwen 等主流模型上均达到 SOTA。

🎯 为什么值得关注？

2026 年初，DeepSeek 系列模型（含 MoE 架构）爆火，带动了 LLM 推理服务需求的巨幅增长。SGLang 是少数能原生高效支持 DeepSeek 的推理框架，并且对 Blackwell（GB200/B300）、AMD MI300、Google TPU、华为昇腾均有适配——这意味着它不只是 NVIDIA 生态的工具，而是一个真正跨平台的生产级推理底座。

  • 🏆 在开源 LLM 推理引擎中持续刷新吞吐量基准
  • 🧠 原创 RadixAttention 被学术界引用，已产生广泛影响
  • 🌐 跨硬件平台（NVIDIA/AMD/TPU/昇腾）统一 API
  • 🔄 与 RLHF/on-policy 训练流程无缝衔接，DeepSeek 官方也在用

🔑 核心技术：RadixAttention 前缀缓存

这是 SGLang 最有原创性的贡献。传统推理框架每次请求都要重新计算 KV Cache，而 SGLang 用基数树（Radix Tree）结构管理缓存，自动识别不同请求的公共前缀并复用。

举例：聊天机器人的系统提示（system prompt）通常几百 token，每次请求都要重复计算。RadixAttention 让这部分只算一次，后续所有请求直接命中缓存，推理速度显著提升。

请求1: [系统提示(500 tokens)] + [用户问A(20 tokens)]
请求2: [系统提示(500 tokens)] + [用户问B(18 tokens)]
             ↑
  RadixAttention 自动共享前 500 tokens 的 KV Cache
  → 节省 95%+ 的前缀计算开销

KV Cache 管理层次

组件功能
RadixCache树形结构，实现请求间自动前缀共享
ReqToTokenPool请求索引 → KV Cache 槽位映射
TokenToKVPoolKV 缓存数据存储管理
ScheduleBatch批次请求元数据与分配信息

🏗️ 多进程解耦架构

SGLang 将整个推理管线拆分为三个独立进程，通过 ZeroMQ (ZMQ) 通信，彻底规避 Python GIL 竞争：

进程负责设计优势
TokenizerManager分词、请求 I/O异步处理，不阻塞 GPU
Scheduler调度、内存管理CPU 密集，与 GPU 完全解耦
ModelRunner实际推理（前向传播）纯 GPU 计算，最大利用率

相比单进程框架：CPU tokenization 不再拖累 GPU 推理，在高并发场景下吞吐量提升可达 30%+

⚙️ 并行策略 & 量化支持

全面的并行方案

  • 张量并行（TP）：单模型切分到多 GPU，适合大参数量模型
  • 流水线并行（PP）：多节点流水作业，跨机器部署
  • 专家并行（EP）：专门针对 MoE 模型（DeepSeek V3、Mixtral）
  • 数据并行（DP）：多实例负载均衡，线性扩展吞吐

量化方案全覆盖

FP8 → MXFP8 → FP4 → MXFP4 → INT8 (AWQ / GPTQ / Marlin)
        ↑ 新 Blackwell 硬件原生支持 FP4/MXFP4，显存减半

投机解码（EAGLE 算法）

通过小草稿模型预测多个 token，大模型批量验证，解码速度提升 2–3×，对长文本生成场景效果尤其显著。

🚀 快速上手

安装

pip install sglang[all]

启动服务（OpenAI 兼容 API）

python -m sglang.launch_server \
  --model-path meta-llama/Llama-3.1-8B-Instruct \
  --tp 2 \
  --port 30000

调用

curl http://localhost:30000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "llama3", "messages": [{"role": "user", "content": "你好！"}]}'

Python 原生 DSL（SGLang 独有）

import sglang as sgl

@sgl.function
def multi_turn_chat(s, prompt):
    s += sgl.system("你是一个专业助手")
    s += sgl.user(prompt)
    s += sgl.assistant(sgl.gen("reply", max_tokens=512))

state = multi_turn_chat.run(prompt="解释量子纠缠")
print(state["reply"])

这个 DSL 让开发者用 Python 语法直接控制推理流程，写复杂 Agent 逻辑比拼接字符串更自然，是 SGLang 区别于 vLLM 的核心体验差异。

⚔️ SGLang vs vLLM：如何选择？

✅ 选 SGLang 的场景

  • 多轮对话 / 共享系统提示（RadixAttention 优势大）
  • 需要自定义推理流程（Python DSL）
  • DeepSeek / MoE 模型部署
  • Blackwell 或 AMD 新硬件
  • RLHF on-policy 训练配套

✅ 选 vLLM 的场景

  • 需要更丰富的插件生态
  • 已有 vLLM 集成的上下游工具
  • 社区支持优先

📅 最新进展（2026 年 4 月）

  • sgl-router：新增路由层，支持多实例负载均衡，方便大规模集群部署
  • Diffusion LLM（DLLM）：开始支持扩散语言模型，拓展至图像生成领域
  • Blackwell 率先适配：原生支持 NVIDIA GB200/B300，FP4 量化性能翻倍
  • LoRA 动态热切换：运行时切换 LoRA Adapter，无需重启服务，多租户场景实用
  • MXFP4/FP8 量化完善：新一代低精度量化方案正式稳定

👥 适合谁用？

  • AI 应用开发者：需要部署私有大模型服务，追求低延迟高并发
  • ML 研究者：做 RLHF / on-policy 训练，需要快速推理服务配合
  • 企业用户：搭建内部 LLM API 网关，需要兼容 OpenAI 接口标准，并希望掌控自己的推理基础设施
